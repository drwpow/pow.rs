You‚Äôve doubtless used CSS <a href="https://developer.mozilla.org/en-US/docs/Web/CSS/transition" target="_blank">transition</a> or <a href="https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_Animations/Using_CSS_animations" target="_blank">animation</a> a million times by now:

```css
a {
  transition: color 100ms;
}
```

Or, if you‚Äôve gotten _really_ fancy, perhaps you‚Äôve specified an easing curve:

```css
a {
  transition: color 100ms ease-in-out;
}
```

But have you ever stopped to consider if it‚Äôs the _right_ easing curve? After all, every animation has _some_ curve; that‚Äôs just one way of representing the motion taking place. Although there‚Äôs no ‚Äúright‚Äù or ‚Äúwrong‚Äù easing curves for every situation, each scenario will strongly lend itself toward a particular animation type for the most responsive, fluid-feeling result.

## what is an easing curve?

_Note: The ‚Äúbible of animation,‚Äù <a href="https://bookshop.org/books/the-animator-s-survival-kit-a-manual-of-methods-principles-and-formulas-for-classical-computer-games-stop-motion-and-internet-animators-expanded/9780865478978" target="_blank">The Animator‚Äôs Survival Kit</a>, illustrates the concept of easing well (as would be expected from Richard Williams). This section is a cheap imitation of the lessons there. I can‚Äôt recommend buying it enough as it teaches the fundamentals to animation in all forms, including web._

This ùë•/ùë¶ representation of the animation is an <b>easing curve</b>. This is the only instance an easing ‚Äúcurve‚Äù¬†will lack the curvy bits, and this is why dividing time/space evenly is called a <b>linear curve</b> (i.e. straight curve). This is simply the easiest to understand; in the next example we‚Äôll see the ‚Äúcurve‚Äù part of it.

All together, these compose the four major types of easing curves‚Äî<b>linear</b>, <b>ease-in</b>, <b>ease-out</b>, and <b>ease-in-out</b>. Most animations use one of these four curves. Of course, we‚Äôve only scratched the surface. The ease-\* curves may be tweaked slightly to provide more nuance to the motion (<a href="https://easings.net/" target="_blank">example</a>). And there are more advanced curves that don‚Äôt fit into one of these categories (such as springs). But you now understand the ‚Äúbread and butter‚Äù of animation curves from these four types alone. So with that out of the way, here‚Äôs how to apply those in practice.

## I. light, color, brightness, and opacity

When animating any properties of light (color, brightness, opacity), you‚Äôll always want to use the <b>linear easing curve</b>.

```css
a {
  transition: color 100ms linear;
}
```

Out of any standard easing curve, a linear one will produce the most even blend of color (that‚Äôs how color is already optimized to work!). Compare a linear transition (top) to an ease-out transition (middle) to an ease-in transition (bottom):

While nonlinear curves may be desirable with motion, they produce uneven blending for colors. You can see how a linear transition (top), for the most part<sup>\*</sup>, blends color A and color B evenly. But when using ease-out (middle), the midpoint gets shifted too far left, and ease-in (bottom) shifts the midpoint too far right. So in most cases, a <b>linear curve</b> is the best option.

Sure, there may be a _super-specific_ usecase where you want to apply an easing curve to a color transition. But that will only ever work with a very specific curve and a very specific color transition‚Äîchange one, and you must change the other (it will never be generally-applicable).

_<sup>\*</sup>If you think linear transitions don‚Äôt transition smoothly, congratulations! üéâ You‚Äôve <a href="https://observablehq.com/@sebastien/srgb-rgb-gamma" target="_blank">found a rabbit hole into color theory</a>. <i>glhf</i>. But for your own sanity, just pretend that linear transitions are the smoothest among ‚Äúdumb‚Äù non-color-aware easing curves. And pretend linear curves are good enough for animating (which, IMHO, they are)._

## II. rotation

Rotation is another good general application for a <b>linear curve</b> (though there may be a few more exceptions here). The chief offender I see is ‚Äústuttering spinners‚Äù (left) that are more common than they should be:

The poor excuse for a spinner on the left, bumping along almost like a flat tire, is what you get with the brower-default <code>ease</code> curve. Inspecting the code you‚Äôll find the fix is as simple as a single CSS property:

```diff
  .spinner--stutter {
+   animation-timing-function: linear;
  }
```

_Note: <a href="https://developer.mozilla.org/en-US/docs/Web/CSS/transition-timing-function" target="_blank">transition-timing-function</a> is the equivalent property for CSS transitions you may be looking¬†for._

Rotation is a bit more nuanced than light, so a linear curve isn‚Äôt an ‚Äúalways‚Äù rule here (a good example is <a href="https://material.io/components/progress-indicators#circular-progress-indicators" target="_blank">Material Design‚Äôs progress indicator</a>, which is a multi-layered animation unifying several curves, but if you look closely, a linear rotation underpins the whole thing). But I‚Äôd still recommend **starting with a linear curve for rotation** and changing it if it‚Äôs not quite right.

## III. interaction

Interactive elements is a broad category of animations, consisting of everything clickable, hoverable, draggable, etc. But despite its generality, here are 3 common mistakes that, if avoided, will probably take care of most animation issues in your UI:

### mistake 1: sluggish interactions

Humans think of an interaction as ‚Äúinstant‚Äù (i.e. <i>not slow</i>) if it happens within ~100ms (<a href="https://www.nngroup.com/articles/response-times-3-important-limits/" target="_blank">source</a>). While <a href="https://material.io/design/motion/speed.html#controlling-speed" target="_blank">animation speed is important</a>, we‚Äôll focus on the easing curve. Compare the two examples of hover, with one using <b>ease-in</b> and the other using <b>ease-out</b>. Which feels more ‚Äúinstant‚Äù?

As a general rule, **ease-out curves are better for interactions because they respond quicker**. The user is able to notice the change more easily because most of the movement happens up-front (scroll back to the ease-out curve earlier in the article as proof). And as it goes with any rule of thumb, this rule can be broken, sure, and there are dozens of scenarios where ease-in or ease-in-out make a better animation. But starting with an ease-out curve for interactions will feel ‚Äúright‚Äù most of the time.

### mistake 2: easing drag events

Drag events should track 1:1 with the user‚Äôs mouse or touch input. Once you start animating _everything_, it‚Äôs easy to fall into the trap adding easing curves for things that shouldn‚Äôt be animated at all. Dragging is the chief offender. ### general recommendation

Again, since interactions are complex and very situational, it‚Äôs hard to develop blanket rules. But hopefully these 3 mistakes get you thinking about how to reason through what works and what doesn‚Äôt in animating each scenario. But when in doubt, defaulting to <b>ease-out</b> will be right for _most_ interactions. Change it if needed (especially if a real-world parallel behaves differently), and never forget that <b>no easing</b> is always an option (as in the dragging example).

## summary

Now that you‚Äôve been made aware of the quick wins, you can apply the following today for a much-improved user experience:

- <b>light</b>: use linear easing
- <b>rotation</b>: start with linear easing by default, but adjust to taste
- <b>interactions</b>: start with ease-out by default, but adjust to taste, and also consider no easing as an option

But there are many more challenges beyond this! And there are more niche, contextual situations to encounter where easing may be more complex. For example, think back to the dragging example above. What if the handle needed to reset back to center after dragging (like a joystick)? You‚Äôd need to remove the easing curve while dragging, but re-apply it as it ‚Äúslid‚Äù back to center position. Keep an eye out for multi-layered easings where one component may require different animations for different states.

In respect of that, some more nuanced, universal approaches to animation easings would be:

- easings help the human brain notice a change (e.g. an object animating across a screen will be better perceived than ‚Äúteleporting‚Äù in one frame). use them to improve recognition of an action
- more movement & change up-front also help with this recognition (ease-out)
- make animations mimic the real world (e.g. all objects have momentum while moving and accelerate/decelerate appropriately; switches ‚Äúflick‚Äù on-and-off and don‚Äôt smoothly glide, etc.)
- animations introduce input lag! So only add them when a user is not dragging/zooming/scrolling that very instant

Play around with your newfound skills and you‚Äôll be making more sophisticated animations in no time. However, you‚Äôre also now cursed with noticing bad animation everywhere you look, so good luck with that, too.
